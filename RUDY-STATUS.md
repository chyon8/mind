## 구현 중 내린 결정 (RUDY-BUILD 대비 변경/확정)

- **링크 임베딩 = 제목+og:description** (요약 아님). 유튜브는 og:description이 홍보문구로 오염돼
  페이지의 `ytInitialPlayerResponse.shortDescription`을 직접 파싱 (linkMeta.ts + 스크립트 동일 로직).
- **다이제스트(LLM 요약·의도) 보류.** 비용이 아니라 — (1) 보여줄 표면 없음, (2) 검색품질은
  설명글 임베딩으로 이미 해결, (3) 의도 해석은 lazy가 §2-1 정합. → 채팅 만들 때 lazy 계산.
- **검색 폴백**: 임베딩/RPC 실패 시 키워드 검색으로 자동 폴백(+console.warn).
- **타입 필터 = 서버(RPC)에서** (클라 후필터 아님).
- 스크립트는 `.mjs` (TS 러너 미설치). embedText/해시는 embed 함수와 반드시 동일.
- **충돌: seed 평균 임베딩 폐기 → seed별 개별 비교 후 최댓값** (RUDY-BUILD B-1에서 변경).
  이유 (1) 무관한 걸 여러 개 던진 날 평균은 노이즈 중심이 되어 억지 충돌을 만든다,
  (2) "왜 지금"(B-3)이 어느 씨앗과 부딪혔는지를 요구하는데 평균은 그 정보를 지운다.
- **충돌 게이트와 순위 분리** (RUDY-BUILD B-2에서 변경). B-2의 `sim×faded×weight ≥ 0.35`는
  weight가 위로 열려 있어(1 + touch_count×2 + …) **자주 만진 파편이 무관해도 통과**하고 평범한
  파편은 관련 있어도 못 통과한다 — 억지 충돌 방지 장치가 정반대로 작동. → 게이트는 `sim`만,
  통과한 것 중 순위만 `sim × faded × weight`.
- **선명도는 SQL로 옮기지 않는다.** 감쇠 법칙의 단일 원천은 vividness.ts. RPC는 "아직 흐려졌을
  리 없는 것"(7일 미만)만 보수적으로 걷어낸다. 진단용 `min_age_days=0` 파라미터로 필터를 끌 수 있음.
- **충돌 임계값 0.35 → 0.42** (실측, `check-collision.mjs --all`). 실제 코퍼스 41개 후보로 보니
  0.35는 깨끗한 경계가 아니었음 — teenage engineering 클러스터는 0.52~0.73으로 확실했지만
  0.30~0.34대에 무관한 것과 관련 있는 것이 뒤섞여 있었다(임베딩 이방성). 억지 충돌보다 놓치는
  쪽이 안전하다는 §2-8에 따라 올림. **진짜 튜닝은 원장(§10-4) 게이트 판정 로그로 — 이 값은 잠정.**

## §10-4 원장 — 구현 범위 (의도적으로 좁힘)

**만든 것:** `rudy.utterances`(원장) + `rudy.gate_log` + 30일 되살리기 쿨다운 + acted/dismissed 캡처.

**안 만든 것 — §6-4의 5개 게이트를 추상 프레임워크로 만들지 않았다.** 지금 말하는 주체가
충돌 회상 하나뿐이라 범용 파이프라인은 speculative abstraction(CLAUDE.md §2)이다. 실제로 쓰이는
게이트만 인라인으로 있다: **유사도 게이트**, **쿨다운 게이트**. 화법·확신 게이트는 생성 발화가
없어서 적용 대상 자체가 없고, 예산은 하루 1개 저장(AsyncStorage day key)으로 이미 강제된다.
→ **두 번째 화자(채팅)가 생길 때 공통 모양이 드러나면 그때 추출한다.**

설계 판단 기록:
- `item_ids`에 **씨앗은 넣지 않는다.** "왜 지금"은 렌더 시점 계산·무저장이고 연결 저장은
  Mind SPEC §7 금지. 씨앗은 기기 로컬(AsyncStorage)에 하루치만 남는다.
- **랜덤 픽은 원장에 안 들어간다.** 그건 루디의 발화가 아니라 우연의 자리(§4-F2)다.
  따라서 랜덤으로 뜬 파편에 기억하기/흘려보내기를 해도 루디 성적에 반영되지 않는다 — 의도된 것.
- `ignored`는 저장하지 않는다. "응답 없음 + 24h 경과"로 집계 시점 계산 (노출은 신호가 아니다 §2-3).
- 원장 기록 실패가 회상을 죽이지 않는다 (try/catch + warn).

## §10-5 채팅 — 구현 중 내린 결정

- **자발적 연결의 원장 kind = `resurface`** (surface만 `chat`). 되살리기와 같은 kind를 쓰면
  기존 쿨다운 조회(`resurfacedIdsSince`는 surface를 안 가림)에 자동으로 걸려서, §4-C1이 요구한
  "표면 간 중복 방지"가 **새 코드 없이** 성립한다 — 떠오른 것에서 본 파편을 채팅이 또 꺼내지 않는다.
- **평범한 채팅 답변은 원장에 안 적는다.** 원장은 §2-2 "먼저 거는 말"의 중복 방지 장치인데,
  질문에 답한 걸 전부 넣으면 반복 게이트가 오염된다(§6-4 ⑤도 채팅 응답은 무예산).
  원장에 남는 건 **묻지 않았는데 나간 자발적 연결 하나뿐.**
- **채팅 충돌 임계 = 0.34** (회상은 0.42). 질문↔파편은 파편↔파편과 유사도 분포가 다르다 —
  질문 임베딩은 문서 임베딩과 비대칭이라 같은 관련도에서도 낮게 깔린다. **완전 잠정값이고,
  판정마다 gate_log에 실측을 남긴다** (`surface='chat'`) — 이번엔 §6-4 "감이 아니라 로그로
  튜닝"이 처음부터 가능하다. 며칠 쓰고 `check-ledger.mjs`로 분포 보고 조정할 것.
- **연결 언급 여부는 검증하지 않는다.** 모델에 `<연결>`을 건네고 "억지면 말하지 마라"라고
  지시하므로, 원장의 의미는 "연결을 건넸다"까지가 정확하다. 언급 강제는 화법을 망친다.
- **자발적 연결은 RAG 근거로 이미 인용된 파편을 제외한다** (`exclude_ids`).
  답에 쓴 걸 "이것도 이어지는데?"라고 가리키면 연결이 아니라 자기 답변 반복이다.
- **원탭 진입 질문에 파편 내용을 풀어 넣는다.** "이거 관련 뭐 있었지"만 보내면 임베딩에 주제가
  없어 RAG가 아무거나 물어온다 → `『{제목}』 관련해서 내가 저장해둔 게 또 뭐가 있지?`로 만든다.
- **채팅은 드로어 탭이 아니라 push 화면**(`src/app/chat.tsx`). 드로어는 피드를 거르는 렌즈들의
  자리인데 채팅은 렌즈가 아니라 목적지다. RUDY-BUILD C-3의 `(drawer)/chat.tsx`에서 변경.
- **대화 기록 시트 있음** (유저 요구, 2026-07-19). 목록·이어가기·보이는 지우기 버튼.
  대화 행은 첫 전송 때 지연 생성 — 열 때마다 만들면 빈 대화가 쌓인다.
- **스트리밍은 `expo/fetch`.** supabase-js `functions.invoke`는 응답을 통째로 모아서 주므로
  못 쓴다. 함수 URL을 직접 치고 세션 토큰을 붙인다. 프로토콜은 NDJSON(`d`/`cite`/`link`/`error`).
- **이력 저장은 서버(Edge Function)가 한다.** 스트리밍 도중 앱이 죽어도 대화가 남아야 한다.

### 1차 실사용에서 터진 버그 3건 (2026-07-19, "답이 나오다가 없어짐")

셋 다 원인이 달랐고 증상이 겹쳐 보였다. 스트리밍 UI 만들 때 재발하기 쉬운 것들이라 남긴다.

1. **스트림 끝의 남은 버퍼를 버렸다.** SSE·NDJSON 둘 다 `split('\n')` 후 마지막 조각을
   다음 청크용으로 버퍼에 남기는데, 루프가 끝날 때 그 버퍼를 안 흘렸다. 공급자가 마지막 줄을
   개행 없이 닫으면 **답변 끝이 통째로 잘린다.** → 양쪽 다 루프 뒤에 flush 추가.
2. **저장 확인 없이 화면을 지웠다.** 스트림이 닫히면 무조건 `streaming=null`로 두고 서버에서
   이력을 다시 읽었는데, 아직 안 적힌 답을 읽으면 **다 나온 답이 눈앞에서 증발한다.**
   → 서버가 저장 후 `done` 이벤트를 보내고, 앱은 그걸 받고 이력에서 확인된 뒤에만 지운다.
   못 믿을 땐 흘러나온 텍스트를 그대로 두고 "저장 안 됨"만 표시한다.
3. **ScrollView에 `flex:1`이 없었다.** 세로 flex 컨테이너 안에서 내용 크기로 자라서,
   답변이 길어지면 입력창을 화면 밖으로 밀어냈다.

**교훈: 스트리밍에서 "사라짐"은 대개 세 곳 중 하나다 — 버퍼 flush 누락, 낙관적 화면 정리,
스크롤 컨테이너 flex.** 셋 다 각각 확인할 것.

### 2차 실사용에서 잡은 진범 (2026-07-19, "다음 전송이 안 됨")

**뿌리는 서버 저장 실패였다 — `check-chat.mjs`로 실측해 확정.**
PostgREST 다중 행 insert는 행마다 키가 다르면 **빠진 키를 default가 아니라 null로 채운다.**
user 행에 `cited_ids`가 없어서 null → not null 제약 위반(23502) → 이력 저장 전체 실패 →
클라이언트가 실패 경로 진입 → `streaming` 상태가 안 풀려 **다음 전송 영구 잠김.**
수정 (1) user 행에도 `cited_ids: []` 명시 — 수정 payload를 실 DB에 직접 쏴서 통과 확인,
(2) 상태머신 자체를 재설계 — phase는 idle↔streaming 둘뿐, `finally`가 무조건 idle 복귀.
**교훈: 클라이언트 상태 잠김의 원인이 클라이언트에 없을 수 있다. E2E 진단부터 만들 것 —
`check-chat.mjs`가 첫 실행에서 바로 이걸 잡았다. 그리고 PostgREST 다중 행 insert는
모든 행에 같은 키를 채워라.**

### 3차 — RAG가 망가져 있던 진짜 이유 (2026-07-19 실측)

**질문 문장을 통째로 임베딩한 것이 원인이었다.** "내가 홈레코딩 관련 저장한 거 있어? 링크도 줘"를
그대로 임베딩하면 "저장/링크" 같은 **메타 표현이 벡터를 지배**해서, 홈레코딩이 아니라
*앱의 링크·저장 기능에 대한 파편*이 상위에 온다:

| 검색어 | 1위 결과 |
|---|---|
| 질문 원문 | 0.535 "링크 던지면 요약해서 왜 저장했는지…" ← 무관 |
| "홈레코딩"만 | 0.511 "음성으로 녹음" ← 정답 |

유저가 검색창엔 "홈레코딩"만 치니까 잘 나왔고, 채팅은 문장을 통째로 던져서 망가졌다.
**같은 RPC를 쓰는데 결과가 달랐던 이유가 이것이다 — 모듈이 다른 게 아니었다.**

→ **질의 재작성(다중 질의 검색)** 도입. 질문에서 주제어 1~3개를 빠른 모델로 뽑고,
주제어별로 검색해 파편별 최고점으로 합친다. **주제어가 나오면 원문은 안 쓴다**(다시 오염됨).
주제어가 없는 질문("요즘 뭐에 꽂혔어?")만 원문으로 검색.
추출 프롬프트에서 동작어(정리/저장/자료)를 명시적으로 배제 — 안 하면 그 단어의 키워드 매치가
1.0점으로 상위를 먹는다. 실측 후 결과: 홈레코딩→음성으로 녹음·기타 코드·vst,
기타→기타 코드 1.000·피아노/기타 연주 1.000. 비용은 재작성 1회(~600ms, gpt-4o-mini).

**교훈: RAG에서 "검색은 되는데 채팅은 안 된다"면 질의 텍스트를 의심하라. 같은 인덱스라도
무엇을 임베딩하느냐가 다르면 다른 시스템이다.**

### 클라이언트 내구성 원칙 (2026-07-19 확정)

**화면에 나온 글자는 서버가 어떻게 되든 사라지지 않는다.** 저장 실패로 텍스트가 증발한
사고 이후 세운 규칙. 로컬 턴 배열(`Turn[]`)에 누적하고, 서버 저장이 확인될 때만 서버 이력으로
승격시킨다. 확인 안 되면 로컬에 남고 "저장되지 않았다"만 덧붙는다.
같은 이유로 **에러를 조용히 삼키지 않는다** — 기록 시트는 조회 실패를 시트 안에 적는다
(예전엔 `.catch(() => {})`라 버튼을 눌러도 아무 일이 안 일어나 "기록이 아예 안 뜬다"가 됐다).

## §10-6 클러스터 엔진 — 구현 완료 (2026-07-20), 배포·검증 대기

**RAG 나열과 다른 지점 (착수 전 정의한 것, 결과물은 이 셋을 만족한다):**
1. **질문이 입력이 아니다** — 질문에 가까운 걸 뽑는 게 아니라 90일 코퍼스에서 서로 가까운
   것끼리 묶는다. 검색이 원리상 못 하는 일.
2. **묶고 이름 붙인다** — 파편 N개가 축 1개로. 나열이 아니라 축소.
3. **시간 모양을 안다** — "그 축, 3주째 조용해"를 말할 수 있다. **없는 것을 말할 수 있는 건
   클러스터뿐이다.**

**구조:**
- `rudy.cluster_edges(days, min_sim, max_edges)` — 쌍 유사도를 SQL에서. 3072차원을 Deno로
  끌어오면 파편 500개에 수십 MB 전송이다. **테이블은 안 만든다** (엣지 목록만 반환).
- `_shared/cluster.ts` — 평균연결 병합 + 선명도 가중 + 시간 모양. 순수 함수, 저장 없음.
- `chat/clusters.ts` — findAxes(계산) + axesBlock(프롬프트 블록). 라벨은 LLM 1콜(fast model).
- 채팅 라우팅: **질의 재작성이 `topics: []`를 뱉는 질문이 정확히 "요즘 뭐에 꽂혔어?" 류다** —
  새 분류기 없이 그 자리를 축 경로로 갈아끼웠다. 예전엔 거기서 질문 원문으로 검색했는데
  그게 3차에서 밝혀진 오염된 경로 그 자체였다. 축이 안 서면 기존 검색으로 폴백(채팅에서 침묵은 답이 아니다).

**구현 중 내린 결정:**
- **평균연결(average linkage)이지 연결요소가 아니다.** 단일연결이면 A–B, B–C 두 엣지만으로
  A·C가 한 덩어리가 된다(체이닝) — 무관한 두 주제가 다리 파편 하나로 붙으면 축이 아니라 죽이고,
  거기 이름을 붙이라 하면 LLM은 반드시 그럴듯한 거짓말을 만든다(§2-8).
- **지속 판정에 "서로 다른 저장일 ≥3"을 추가**(span ≥21일과 함께). span만 보면 하루에 몰아
  저장하고 한 달 뒤 1개 더 저장한 게 "3주간 반복된 축"이 된다 — 그건 반복이 아니라 **두 번
  있었던 일**이다. §4-B1이 보려는 건 흩어짐이지 양끝 거리가 아니다.
- **TTL 캐시 안 만들었다.** §6-2는 TTL 캐시를 허용하지만, 트리거되는 질문이 드물고 라벨링이
  1콜(~1.5s)이다. 캐시가 없는 쪽이 §2-1에 더 안전하고 코드도 없다. 지연이 아프면 그때 만든다.
- **축 경로에선 자발적 연결을 안 돌린다.** 답변 전체가 이미 "묻지 않은 것을 꺼내는" 일이라
  거기 연결을 또 얹으면 같은 동작의 반복이다.
- **이름을 못 붙인 묶음은 버린다.** 라벨 프롬프트가 잡동사니면 빈 문자열을 내게 하고, 빈 것은
  축에서 제외 — LLM에게 억지 명명을 시키지 않는 안전장치.
- **판정을 gate_log에 남긴다** (`kind='cluster'`). 충돌 임계를 감으로 정했다가 뒤집은 것과 달리
  이번엔 처음부터 근거가 쌓인다.

**⚠️ 임계 0.42는 5일치 코퍼스로 정한 잠정값이다.** 실측 근거(2026-07-20, `check-clusters.mjs`):

| 임계 | 축 | 묶인 파편 | 최대축 |
|---|---|---|---|
| 0.34 | 8 | 28 | 5 |
| 0.38 | 4 | 15 | 5 |
| **0.42** | **3** | **11** | **5** |
| 0.46 | 2 | 8 | 5 |

0.42에서 선 축: teenage engineering(5) / 독서·문학(3) / 앱 UI 메모(3). 앞 둘은 깨끗하고
셋째가 제일 느슨하다. 0.46이면 셋째가 사라지고 커버리지가 절반이 된다.

**⚠️ 셋 다 "단발"로 나왔고 그게 정답이다.** 코퍼스 전체가 2026-07-16~20의 5일치다

### 1차 실사용에서 잡은 것 (2026-07-20)

**축 답변 자체는 설계대로 나왔다** — 나열이 아니라 축 4개(믹서/케이스랩/독서/앱UI), 그리고
**"지속적인 관심으로 보이지는 않는다"**고 스스로 말했다. 5일치 코퍼스에서 경향을 지어내지
않은 것 = §2-8이 실제로 작동한 첫 사례.

버그 1건 — **미니 마크다운 파서가 중첩을 못 견뎠다.** 모델이 축을 나열할 때 링크를 굵게
감싼다(`**[제목](mind://…)**`). 인라인 정규식의 `\*\*([^*]+)\*\*`가 링크 마크업을 통째로
삼켜서 **마크업 원문이 화면에 그대로 찍히고 탭도 안 됐다.**
→ 굵게 안을 재귀 파싱하도록 고치고 회귀 테스트 추가(`__tests__/markdown.test.ts`).
**교훈: 프롬프트로 모델의 마크업 습관을 막으려 하지 마라 — 파서가 견뎌야 한다.**
DB 원문을 먼저 확인한 게 진단을 갈랐다 (저장된 URL은 멀쩡했으므로 서버가 아니라 렌더러다).

화법 1건 — 시간 모양이 **계산된 사실**로 넘어가는데 모델이 "생각을 모은 것 같아"처럼
추측 어미를 붙였다. 프롬프트 `## 축`에 금지 문구 추가.

## §10-6+ 늦은 의도 (F1) — 구현 완료 (2026-07-20), 배포·검증 대기

`rudy.evidence`가 **§2-1 경계선의 물리적 실체**다. 루디의 추론은 어디에도 저장 안 하지만
유저가 직접 말한 것은 저장한다. "이 사람은 케이스랩에 빠져 있다"(루디의 결론) ❌ /
"케이스랩 준비하고 있어"(유저의 말) ✅.

**세 갈래를 안 만들었다.** 유저는 "설명 없는 링크 / 고아 파편 / 축의 의도 불명 셋 다 필요"라고
했지만, 실제로 다른 건 **정렬 키뿐**이라 후보 풀 하나 + 우선순위로 짰다:
① 축에 묶였는데 의도 불명(답 하나가 축 전체를 추측→확인으로) → ② 설명 없는 최근 파편
(하나만 설명하지만 최근이라 유저가 실제로 기억한다) → ③ 오래된 고아(물어도 답을 모른다).
트리거 3개·게이트 3개를 만들 뻔한 걸 하나로 줄였다.

**구현 결정:**
- **질문은 채팅 답변 끝에 붙인다.** 새 표면 없음 — §4-F1이 "대화 중 자연스럽게"라 못 박았고
  푸시로 만들면 §2-6 예산·§7-3에 걸린다. 프롬프트가 "흐름에 얹기 어색하면 묻지 마라"까지 지시.
- **쿨다운은 원장 재사용.** `kind='question'` + `item_ids=[파편id]`로 적으면 같은 파편 재질문이
  자동으로 막힌다 — 자발적 연결이 `resurface` 원장을 재사용한 것과 같은 수법. 새 코드 없음.
- **예산 검사를 제일 먼저 한다.** 하루 1개를 이미 물었으면 나머지 조회를 아예 안 돈다 —
  비싼 일이 하루 한 번만 돌게 하는 게 코드 순서로 강제된다.
- **저장되는 문장은 유저의 말 그대로.** 모델은 "이게 답인가"만 판정하고 문장은 안 만진다.
  요약·정규화하는 순간 그건 유저의 진술이 아니라 **루디의 해석**이고 §2-1을 넘는다.
  → `JUDGE_SYS`는 `{"isAnswer": bool}`만 뱉는다. 이 설계를 바꾸지 말 것.
- **소비처까지 같이 붙였다** — 안 그러면 아무도 안 읽는 데이터를 쌓는 거다. 축에 묶인 파편에
  자기 진술이 있으면 `<축>` 블록에 "본인 진술"로 들어가고, 프롬프트가 거기엔 추측 화법을
  금지한다 → §4-B2가 요구한 "확인해주면 추측을 벗는다"가 성립.
- `embedding`은 지금 소비처가 없는데도 채운다. 나중에 §6-4 ② 반복 검사·의미 검색에 필요하고,
  안 채우면 백필을 또 돌려야 한다 (임베딩 백필 이미 두 번 했다).

### F1 1차 실사용에서 잡은 진범 (2026-07-20) — 라우팅 조건이 너무 넓었다

유저가 늦은 의도 질문에 **"그냥 재밌어보여서 그리고 나 음악 했었어 원래"**라고 답했더니
루디가 그걸 질문으로 받아 **축 보고서를 다시 냈다** — 앞 턴과 거의 같은 답(§2-2 위반).
gate_log가 그대로 보여줬다: `question/confidence passed=true`(캡처 성공) 바로 뒤에
`cluster/cluster passed=true`(축 경로 재진입).

**원인: 축 경로 조건을 `topics.length === 0`으로 잡은 것.** 주제어가 안 나오는 건 메타 질문만이
아니다 — 진술·감상·인사·잡담이 전부 빈 배열이다. 3차에서 "질의 텍스트를 의심하라"를 배워놓고
같은 자리에서 프록시 신호를 질문 판정에 재사용했다.

수정 2건:
1. **질의 재작성에 `intent` 필드 추가** (`trend` | `other`). 축으로 가려면 `intent==='trend'` +
   주제어 없음 + **답변 턴이 아님**, 셋 다 만족해야 한다.
2. **`captureAnswer`가 캡처 사실을 돌려주고 `<방금답함>` 블록으로 모델에 전달.** 저장만 하고
   대화에 반영을 안 하니 루디가 자기가 물어놓고 답을 못 들은 것처럼 굴었다. 프롬프트에
   "짧게 받아라, 보고서 금지, 고맙다고 하지 마라, 다시 캐묻지 마라" 명시.

**교훈: "이 메시지가 무슨 종류인가"를 부산물로 판정하지 마라.** 주제어 추출의 빈 배열은
"메타 질문"이 아니라 "구체 명사 없음"일 뿐이다. 판정이 필요하면 판정을 시켜라.

**미해결로 남긴 것:** 축 답변에 반복 게이트(§6-4 ②)가 없다. 같은 질문을 이틀 연속 하면 같은
축 목록이 그대로 다시 나온다. 이번 버그는 라우팅 수정으로 막혔지만 원인이 다른 재발 경로다.

### 시간 질의가 통째로 없었다 (2026-07-20) — "오늘 저장한 거 없다"고 답한 사건

유저: "오늘은 무슨 생각을 했지" → 루디: "아직 오늘 남긴 게 따로 보이진 않네."
**실제로는 그날 6개를 저장했다.** 임베딩 누락 아님(살아있는 파편 119개 전부 임베딩 있음 — 확인함).

**원인 1 — 시간 질의를 검색으로 처리했다.** 파이프라인에 날짜 필터가 어디에도 없다.
"오늘 뭐 저장했지"가 주제어 없음 → 임베딩 유사도 검색으로 갔고, 그날 저장한 게 케이스랩
메모라 질문 문장과 의미가 안 닿아서 안 나왔다. **유사도로 시간을 물으면 영원히 이렇게 된다.**
→ `period` 추출(today|yesterday|week|month) + **유사도를 아예 안 쓰는 날짜 조회 경로**(`<기간>`).
"검색 결과가 아니라 전부다"를 블록에 명시 — 그래야 모델이 "없다"를 단정할 수 있다.

**원인 2 — 타임존.** 서버 UTC / 유저 KST(+9)인데 `systemPrompt`의 오늘 날짜를 UTC로 넣었다.
**KST 00:00~08:59엔 루디가 어제를 오늘로 안다.** 날짜가 "어떤 건 맞고 어떤 건 틀린" 이유.
→ `_shared/time.ts` (KST 상수 고정, 유저 1명). 기간 경계도 KST 자정 기준.
**이 버그는 하루 9시간만 나타나 수동으로는 못 잡는다 → `__tests__/time.test.ts`로 못 박았다.**

**원인 3 — 프롬프트 예시 복사 재발.** 늦은 의도 예시를 `"그러고 보니 어제 던진 『○○』…"`로
썼더니 모델이 17일 파편을 **"어제"**라고 불렀다. 이건 이미 §10-5 4차에서 겪은 실패 모드
(`[『내용 요약』]` 자리표시자 복사)와 같은 것인데 재발시켰다.
→ 예시에서 상대 날짜를 빼고 "블록의 날짜와 오늘을 빼서 말해라, 헷갈리면 숫자로"로 교체.

**원인 2의 나머지 절반 (같은 날 이어서 발견).** 범위만 KST로 고치고 **표시는 UTC로 뒀다.**
그래서 "오늘 7개" 헤더 아래 `날짜: 2026-07-19`인 줄(08:48 KST 저장분)이 섞였고, 모델은
합리적으로 **6개**라고 답했다 — 모델이 틀린 게 아니라 준 데이터가 모순이었다.
→ `kstDate()`를 만들어 파편 날짜를 찍는 **모든 곳**에 적용: `fragBlock`(근거) · `axesBlock`(축) ·
`questionSubject`(늦은 의도) · `shape()`의 저장일 카운트(UTC로 세면 지속 판정이 부풀려진다).
**교훈: 시간대 버그는 한 곳만 고치면 반쪽이 남는다. 경계(자르기)와 표시(찍기)는 다른 코드다.**

**교훈 셋:**
1. **프롬프트 예시에 구체값(날짜·제목·수치)을 쓰면 모델은 그 값을 복사한다.** 이 프로젝트에서
   두 번 터졌다. 예시엔 형식만 쓰고 값은 "블록에서 가져와라"로 지시할 것.
2. **유사도 검색은 시간·개수·존재 여부를 답할 수 없다.** "있냐/없냐", "몇 개", "언제"는
   전부 다른 종류의 질의다. RAG가 만능인 것처럼 라우팅하면 태연한 거짓말이 나온다.
3. 축 답변도 같은 병이 있었다 — 90일 창으로 계산해놓고 "이번 주엔"이라고 말했다.
   → 프롬프트에 "축은 90일 전체에서 계산된 것, 기간을 지어내지 마라" 추가.

## ⚠️ 발견의 렌즈 — 2026-07-20 유저 구술로 확정 (이게 없어서 하루를 날렸다)

**§4-E가 "렌즈 = 내 파편에서 계산된 취향"이라고만 쓰고 실체를 안 정해놨다.** 그래서 구현이
파편을 flat하게 읽고 헛것을 가져왔다. 아래는 유저가 직접 말한 것 — 추론이 아니라 진술이다.

**소스 취향:** Hacker News / Indie Hackers / Product Hunt 결.
**기술 수위:** 적당히 기술적. **너무 기술적이거나 학술적인 건 안 본다** (논문·리서치 링크 X).
**아이디어 확장의 형태:** "비슷한 프로덕트가 있고 **실제로 돈이 된다**" — 실물 + 수익 증거.
개념 설명이 아니라 **누가 만들었고 얼마 버는가**.
**다른 분야:** 예술 자체가 아니라 **새로운 관점**.
**그 외:** 진행 중인 프로젝트에 필요할 기술, 나중에 궁금해할 것, resurface 약간.

⚠️ **6~8은 상한이지 목표가 아니다.** 목표로 두면 채우게 되고 5번째부터 쓰레기가 섞인다 —
좋은 게 4개뿐인 날은 4개다 (§2-8 침묵 기본값).

**→ RUDY.md와 충돌한다. 문서를 고쳐야 함:** §4-E2 `주 2~3회 배치, 3~5개`,
§7-3·§4-F4 `브리핑 하루 1회 최대 3항목` → 대략 10배 볼륨이다. 조용히 바꾸지 말 것.

### 3차 브리핑 실패에서 나온 규칙 (2026-07-20) — 실질 수확 1/6

유저 판정: `5번 같은건 쓰레기야` / `앨범은 내 취향이랑 안맞으니까 빼. 내가 알아서 찾아` /
`프로덕트가 2개라는데 어딨어` / `1번은 접근은 괜찮아. 내용이 별로`.

- ⚠️ **"내가 알아서 찾는 영역"엔 들어가지 않는다.** 음악 앨범 추천이 그랬다. 취향 불일치보다
  큰 얘기다 — **유저가 이미 잘 찾는 영역에서는 루디가 무조건 유저보다 못하다.**
  루디의 가치는 **유저가 안 찾아볼 곳**에 있다. → 제외 영역 목록을 렌즈에 유지할 것.
- ⚠️ **와일드카드는 실전에서 쓰레기가 됐다 — RUDY.md §4-E2 수정 사안.** 스펙은
  *"와일드카드 1개는 축을 끈다. 순수 세렌디피티. '이건 네 축이랑 무관해. 그냥.'"* 이라고
  규정하는데, 그대로 실행하니 **연결 없는 잡학**이 나왔다(심해 금색 구체). 무관함 자체는
  가치가 아니다. 폐기하거나 "먼 연결"로 재정의해야 한다.
- **일반론 금지.** 포폴 항목이 `헤딩이 중요하다`, `양보다 질` 같은 검색하면 나오는 조언이었다.
  유저의 실제 파편으로 구체화하지 않으면 접근이 맞아도 내용이 빈다.
- **구성 라벨을 세어보고 쓴다.** "프로덕트 2 · 항목 7"이라고 썼는데 실제로는 프로덕트 1 · 항목 6.
- **"추가"를 "교체"로 읽지 마라.** 유저는 다른 갈래를 *추가*하라고 했는데 프로덕트를 4→1로
  깎았다. 그 결과 잘 되던 쪽까지 같이 사라졌다.

  `글감`(에세이 소재)은 완전히 다른 것이다. flat하게 합치면 안 된다.
- 후보 → 렌즈 적용 → **버린 것 기록**(§2-b 보이는 거절, 유저가 명시적으로 좋아함)
- 원장 기록으로 반복 방지, resurface 후보(감쇠 + 지금 상황과의 관련성)

**기계화가 어려운 부분 — 여기가 진짜 일이다:**
1. **저장된 북마크 × 프로젝트 대조.** 통한 항목 ①은 `Cluely` 북마크가 뭔지 검색으로 알아내고
   `No phone`의 설명(STT 미팅 어시스턴트)과 겹쳐서 **"같은 물건이네"를 알아챈 것**이다.
   파편만으로도, 검색만으로도 안 나온다. **둘을 겹쳐야 나온다.**
2. **검색 실패 시 각도 전환.** 6번 검색 중 **3번이 재시도**였다 (영어권 STT → 한국 벤더 /
   소상공인 시장 → 국내 1인개발 수익 후기). §4-E2가 "장소 탐색이 실제 승부처"라고 한 게 이거다.
   원샷 검색이면 이 항목들은 전부 실패로 남았다.

**항목 하나의 모양:** 제목(주장) → 어느 파편에서 출발했는지 → 실물·숫자 → **함의 한 줄**
(추측이면 "이건 내 추측" 명시) → 출처 링크 2~4개.

## Exa 판정 — 손으로 돌려보고 나온 결론 (2026-07-20)

**막힌 지점 2개, 둘 다 Exa가 푸는 문제가 맞다:**
- 매 검색마다 리스티클이 절반 이상 (`8 Best ...`, `10 Best ...` 어필리에이트)
- **본문을 못 읽어서 제목만 보고 버렸다** — Product Hunt 항목에서 `Context.dev·Pazi는
  이름만 나오고 뭔지 모르겠다`고 쓴 게 그 증상이다. 검색엔진 요약만 보고 판단하고 있었다.

**Exa가 못 고치는 것 = 재시도 판단.** 위 "각도 전환"은 API가 아니라 루프 설계다.
그리고 이날 헛발질 4건(caselab을 법률로, No phone을 미니멀폰으로, 관심사 flat, STM32 우선순위)은
**전부 데이터를 안 읽어서 생긴 것이고 Exa가 고쳤을 건 0개다.**
→ **결론: Exa는 필요하지만 병목이 아니다. 루프부터 짜고 붙인다.**

## ~~E1 아이디어 발견~~ — 폐기 (2026-07-20)

**유저가 쓸모없다고 판정했고 맞는 판정이다.** `ideas.ts`는 유저의 메모끼리 조합해서
아이디어를 만들었는데, §4-E가 절대 구조로 못 박은 문장이 *"내 것에서만 돌면 에코챔버다"* 였다.
그 문장을 코드 주석에 그대로 베껴 적어놓고 정확히 그 에코챔버를 만들었다.

**왜 그걸 골랐나 (기록용):** E2는 검색 API 키 결정이 필요해서 진행이 막히고, E1은 혼자 끝낼 수

근거를 나중에 갖다 붙인 것.** 선택지를 줄 때 E1을 "권장"으로 올린 것도 그래서다.

**교훈: 막히면 우회하지 말고 막혔다고 보고할 것.** 우회해서 만든 건 버려진다.

삭제: `supabase/functions/chat/ideas.ts`, `supabase/rudy-ideas.sql`, index.ts의 `intent:'idea'` 경로.
⚠️ `rudy-ideas.sql`을 이미 SQL Editor에서 실행했다면 `utterances.variant` 컬럼과
`rudy.idea_similarity` 함수가 DB에 남아 있다. 무해하고, 브리핑 구현할 때 재사용한다

### 발견 브리핑 엔진 — 완성 (2026-07-21)

**전 과정이 `check-brief.mjs`에서 실측 검증됨.** 실제 브리핑 6항목 + 버린 것 + 못 찾은 것,
$0.042. 손 퀄보다 나았다(Exa가 MosaicVoice·Selectd 같은 실물 경쟁자를 물어옴).

- **판단은 모델빨** (§8-1 실측): gpt-4o는 각도가 얕고 예시를 복사, **gpt-5.5**라야 겹치고 합침.
  → `DISCOVERY_MODEL`(기본 gpt-5.5), 채팅(gpt-4o)과 분리. 조립도 같은 모델.
- **검색 = Exa REST 직접**(`discovery/search.ts`, SDK 안 씀). `x-api-key`, type=auto, highlights.
  본문 발췌가 와서 "제목만 보고 버림" 문제 해결. ~$0.007/검색, 하루 8검색 월 ~$1.7.
- **반복 방지**(§6-4 ②, 유저가 걱정): 최근 30일 브리핑 URL을 원장 text에서 뽑아 조립 프롬프트에
  `<이미 브리핑함>`으로 넣는다. 새 게이트 없이 — 자발적 연결이 원장 재사용한 것과 같은 결.
- **수익 숫자 요구 완화** (유저: "실제 수익 안 나와도 됨") — 웹에 그런 숫자 잘 없다, 억지 금지.
- 코드: `discovery/{material,angles,search,brief,index}.ts` + node 미러 `scripts/_discovery-lib.mjs`
  + 진단 `check-angles.mjs`·`check-brief.mjs`.

**⚠️ 배포 전제 — Supabase 시크릿:** `EXA_API_KEY` 추가 필요(로컬 .env엔 있지만 Edge Function은
시크릿을 쓴다). (선택) `OPENAI_DISCOVERY_MODEL=gpt-5.5`. `OPENAI_API_KEY`는 이미 있음.
**Deno 런타임·시크릿은 로컬 검증 불가 — 배포해야 확인된다** (node 검증은 로직까지만).

### E1 아이디어 발견 — 구현 완료 (2026-07-20)

스펙이 **최우선**으로 지정한 것(§4-E 우선순위: 아이디어 > 영상 > 책 > 긴 글)이고,
외부 API가 없어도 되는 유일한 조각이다 — 아이디어는 검색이 아니라 **내 축에서 생성**하니까.

- **표면 = 채팅, 요청형.** 질의 재작성의 `intent`에 `idea`를 추가했다("아이디어 줘",
  "뭐 만들면 좋을까"). 피드(§7-4)가 아직 없으므로 자발적 발화는 안 넣었다.
- **5종 로테이션** cross·gap·next·revive·counter. `utterances.variant`에 기록하고 다음번엔
  그 다음 타입부터 — "X × Y 어때"만 반복하면 5번째부터 공식으로 읽힌다(§4-E1).
  **재료가 없는 타입은 건너뛴다** (교차는 살아있는 축 2개, 부활은 가라앉은 축 1개+살아있는 축 1개).
- **살아있는 축 / 가라앉은 축은 평균 선명도로 가른다** (합계로 하면 큰 축이 무조건 살아있게 나온다).
  → **부활 조합(④)은 코퍼스가 늙어야 켜진다.** 지금은 5일치라 가라앉은 축이 0개다.
  RUDY.md가 "감쇠 데이터를 가진 Mind만 할 수 있는 유일무이한 무기"라고 쓴 게 이건데,
  데이터가 자라야 작동한다 — 지속 축(§4-B2)과 같은 상황이다.
- **게이트 2개를 실제로 건다.** ① 근거: 본문에서 `mind://fragment/<id>`를 되읽어 **2개 미만이면
  폐기**(§6-4 ① 패턴·예측류 ≥2). ② 반복: 생성한 아이디어를 임베딩해 과거 아이디어와 cos > 0.85면
  폐기(§6-4 ②). 비교는 `rudy.idea_similarity` RPC로 — 3072차원을 앱으로 끌어오지 않는다.
- **아이디어는 본 모델로 만들고 그대로 흘린다.** 빠른 모델로 만들면 뻔해지고, 만든 걸 본 모델에
  다시 태우면 화법이 뭉개지고 근거 링크가 흔들린다 → `preAnswer` 경로로 조각내 스트리밍.
- **아이디어 경로에선 늦은 의도를 묻지 않는다.** 완성된 글을 그대로 흘리므로 `<물어볼것>`이
  모델에 도달하지 않는데 원장엔 "물어봤음"이 적혀 **그 파편이 영영 잠긴다.** 실제로 넣을 뻔했다.
- 생성 실패·게이트 폐기 시 **null → 일반 경로로 흘러간다.** 채팅에서 침묵은 답이 아니다.
