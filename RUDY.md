# RUDY.md — 기획서 (설계 확정 전 초안, 개발 착수용)

> Rudy는 Mind 파편 저장소 위에 얹는 **출력 층(층 3)**이다.
> 저장소가 "기억"이라면 Rudy는 그 기억을 뒤져서 유저를 돕는 어시스턴트다.
> **Rudy는 유저 대신 생각해주는 역할을 한다.**
>
> Mind SPEC.md와의 관계: Mind 데이터의 규칙(선명도·touch·저장 원칙)은 SPEC.md가 이긴다.
> Rudy의 행동 규칙은 이 문서가 정의한다.

---

## 1. 구조 — 3층

| 층 | 이름 | 역할 |
|---|---|---|
| 1 | 저장소들 | Mind 파편 + 캘린더 + 투두 + 유튜브 나중에 볼 영상 + (확장 가능) |
| 2 | 통합 인덱스 | 모든 소스를 하나의 `item`으로 정규화 + 임베딩 |
| 3 | Rudy | 인덱스를 읽고 유저에게 말하고 행동하는 어시스턴트 |

층 1의 각 소스는 **어댑터**로 붙는다. 어댑터는 읽기 전용이 기본이며,
쓰기(투두 추가, 캘린더 이벤트 생성 등)는 액션 프레임워크(§4-D)의 승인 게이트를 통해서만 한다.

## 2. 절대 원칙 (기능 결정과 충돌 시 이쪽이 이긴다)

1. **규정 금지 — 유저에 대한 결론을 저장하지 않는다.**
   "이 사람은 이런 사람이다"를 어디에도 적지 않는다. 지금 나타나는 패턴은 언제든 바뀔 수 있고
   정확하지 않을 수 있다. 인간의 성격은 항상 변하고 흐른다. 유저를 규정짓고 룹에 가두는 것이
   Rudy가 하지 말아야 할 것 중 최악이다.
   - 저장하는 것은 **증거**(파편, 유저의 직접 진술, Rudy 자신의 발화 기록)뿐이다.
   - 관심사·경향·프로젝트 추정은 **물어볼 때마다 새로 계산**한다 (Mind가 선명도를 저장하지 않고
     매번 계산하는 것과 같은 원리 — "데이터는 사실만, 해석은 화면에서").
   - 패턴은 증거의 감쇠를 상속한다. 유저가 어떤 주제를 떠나면 그 주제는 증거가 안 쌓여서
     **자연히 계산 결과에서 빠진다.** 지워줄 필요조차 없어야 한다.
   - 화법도 규정 금지를 따른다: 항상 시간 한정("요즘 3주간 ~가 자주 보여"),
     절대 정체성 단정("너는 ~한 사람이잖아") 금지.
2. **같은 말 금지.** 추천했거나 제안했거나 했던 말을 반복하지 않는다.
   모든 발화를 원장(§5)에 기록하고, 말하기 전에 과거 발화와 유사도를 검사한다.
   제안했던 내용도 기억해두고 반복하지 않는다.
3. **노출은 touch가 아니다** (Mind SPEC §5-1 상속). Rudy가 파편을 읽거나 띄운 것은
   아무 신호가 아니다. 유저의 판단(열기·기억하기)만이 touch다.
   **구현 강제: Rudy의 모든 읽기 경로는 `last_touched_at`을 절대 갱신하지 않는다.**
4. **모르거나 확실하지 않으면 무조건 질문 후 답변.** 추측으로 단정하지 않는다.
   확신 없는 발화는 관찰·질문 형태로 바꿔서 내보낸다.
5. **발견은 1급 기능이다 — 단, 미는 게 아니라 당기는 표면으로.** 새 발견(§4-E)은 되떠오름과 대등한 핵심이다.
   되떠오름이 내가 **가진 것**을 보여준다면 발견은 내가 **아직 없는 것을 바깥에서** 물어온다.
   자기 표면('발견' 피드, §7-4)을 가지되, 산발 푸시는 금지 — 당기거나 아침 브리핑에 1개까지.
6. **발화 예산.** Rudy가 먼저 거는 말에는 상한이 있다(§6-4). 스팸이 되는 순간 죽는다.
7. **입력 마찰 0 유지** (Mind SPEC §2 상속). 저장 시점에 Rudy가 아무것도 묻지 않는다.
   의도·맥락이 궁금하면 나중에 대화에서 가볍게 묻는다(§4-F1).
8. **침묵 기본값.** 말할 가치가 임계를 못 넘으면 그 슬롯은 비운다 — 억지 충돌·억지 발견·빈 브리핑 금지.
   침묵은 실패가 아니라 신뢰 저축이다. wow는 보장하는 게 아니라 확률을 높이는 것이고,
   없는 날은 없다고 말하는 것이 설계다.

### 2-b. Rudy의 목소리 (캐릭터)

게이트(§6-4)가 **무엇을** 말할지 정한다면, 목소리는 **어떻게** 말할지 정한다.
GPT류가 공허한 이유의 절반은 지식 부족이 아니라 캐릭터 부재다.
- 짧게 말한다. 아첨하지 않는다. 확신 없으면 묻는다(§2-4).
- **취향이 있다. 별로면 별로라고 한다.** 모든 걸 좋다고 하는 추천자는 신뢰가 0이다 —
  큐레이터의 신용은 거절에서 나온다.
- **보이는 거절**: 발견 피드에 가끔 한 줄 — "○○도 떴는데 리스트클이라 버렸어."
  필터가 일하는 게 보일 때 통과한 것들의 가치가 올라간다 (후보 로그는 TTL 캐시, 결론 아님).

## 3. 지식 베이스 (층 1 + 층 2)

### 3-1. 소스와 어댑터

| 소스 | 접근 | 상태 |
|---|---|---|
| Mind 파편 | 같은 Supabase, 직접 읽기 | 확정. 스키마 완전 통제, 첫 어댑터 |
| 캘린더 | Google Calendar API (읽기 + 이벤트 생성은 액션 경유) | 확정 (Google) |
| 투두앱 (Things) | ⚠️ 공식 API 없음. **읽기 = 맥 브리지**, **쓰기 = URL scheme** (아래) | 확정 (우회 설계) |
| 유튜브 나중에 볼 영상 | ⚠️ YouTube Data API는 Watch Later 접근 불가 (2016 폐쇄) → **공유 시트로 Mind에 던지기로 일원화** (결정 2026-07-18). 유튜브 어댑터는 만들지 않는다 | 확정 |

**Things 어댑터 설계 (공식 API 부재의 우회):**
- **읽기**: Things 맥 앱의 로컬 SQLite DB를 읽어 Supabase `items`로 밀어올리는 브리지 스크립트.
  맥에서 launchd로 주기 실행 (예: 30분). 맥이 꺼져 있으면 데이터가 그 시점에 멈출 뿐 — 치명적이지 않음.
- **쓰기 (투두 추가)**: Things URL scheme (`things:///add?...`) — iOS에서 앱 간 호출로 가능.
  액션 프레임워크(§4-D1)의 승인 게이트를 거쳐 실행.

### 3-2. 정규화 스키마 (층 2)

```
items                 -- 모든 소스의 통합 뷰
  id, source (mind|calendar|todo|youtube|...), source_id,
  kind (fragment|event|task|video|...),
  content, created_at, occurs_at (이벤트/할일의 시점, nullable),
  metadata jsonb      -- 소스별 원본 필드
  -- mind item은 선명도 계산 입력(last_touched_at, tier, touch_count)을 그대로 노출

item_embeddings       -- pgvector
  item_id, embedding vector, embedded_at
```

- Mind 파편은 복사하지 않고 뷰로 노출한다 (단일 진실 원천 유지).
- 임베딩은 **저장 시점에 비동기 생성** (DB trigger → Edge Function). 유저는 아무것도 기다리지 않는다.
- **링크 파편은 URL 문자열로 임베딩하면 벡터가 무의미하다 (유령 파편).** 다이제스트(§4-B3) 완료 후
  **요약 텍스트로 재임베딩**한다. 유튜브 공유시트 일원화로 링크 비중이 높으므로, 다이제스트는
  부가 기능이 아니라 임베딩 파이프라인의 인프라다 (§10 순서 반영).
- 기존 파편 전체 백필 1회.
- 임베딩 모델은 한국어 성능 필수 (한국어 파편이 대부분).

## 4. Rudy가 하는 일 — 전체 기능

> ⚠️ 모든 발화형 기능은 §6-4 발화 파이프라인(근거→반복→화법→확신→예산)을 통과한 것만 나간다.

### A. 되살리기 (resurface)

**A1. 되떠오름 — 충돌 엔진.** 가라앉아 흐려지던 파편 중 **오늘 던진 것과 부딪히는** 것을
수면 위로 끌어올린다. Mind의 랜덤 회상을 의미 기반으로 업그레이드하는 것.
Mind SPEC §8-14가 예정한 "AI가 무엇을 언제 떠올려줄지 고르는 역할" 그 자체.
- 후보 = 선명도 0.5 이하(잊히기 직전) + 쿨다운 통과(같은 파편 30일, 흘려보낸 것 60일 — Mind 상속)
- 오늘 파편과의 유사도 × 흐려진 정도 × tier/touch_count로 점수화
- 출력은 Mind **떠오른 것** 슬롯으로 공급 (§7-1). **억지 충돌 금지(§2-8)** — 점수 임계를 넘는 날만
  1개 충돌 + 1개 랜덤(§4-F2), 못 넘는 날은 랜덤 2개. 충돌은 일일 할당량이 아니다 —
  약한 매칭 몇 번이면 유저는 무시를 학습하고 마법 축 전체가 죽는다.
- **요청 시 가시성**: 떠오른 파편을 탭/롱프레스하면 "오늘 던진 ○○와 부딪혀 떠올랐어"를 보여준다.
  렌더 시점 계산, 무저장 (Mind §7 연결 저장 금지와 정합). 유저 1명 제품에서 보이지 않는 개선은
  개선 없음과 구별되지 않는다 — 평소엔 조용, 궁금하면 읽힌다.

**A2. 맥락 리서페이스 — 시간·장소·계획 트리거.** 리스트업해둔 것들 중 시간·장소·계획에 따라
할 만한 게 있으면 되살린다.
- 캘린더 트리거: "여행" 이벤트 감지 → 여행 프로젝트/여행 의미 클러스터의 파편을 출발 전에 상기
- 시간 트리거: 아침 브리핑(§4-F4)에 실림
- 장소 트리거: 지오펜스 (Expo location) — 저장해둔 장소 관련 파편 근처에서 상기
- "여행 준비를 하는 것 같다"의 감지 = 캘린더 이벤트 or 최근 파편 클러스터. 확신 없으면 §2-4에 따라 질문
  ("다음주 제주 가는 것 같은데, 맞아? 저장해둔 것 몇 개 꺼내줄까?")

**A3. 행동 넛지.** 계속 흐려지기만 하고 **한 번도 안 건드린** 파편을 짚는다.
"이거 3주째 안 건드렸는데, 버릴까 진짜 할까?"
- 후보 = touch_count 0 + 바닥 근접 + 원래 신호가 있던 것(링크·행동형 문장)
- **반드시 질문형.** 판단은 유저가 한다. 응답 버튼은 Mind의 기억하기/흘려보내기를 그대로 재사용
  (새 개념을 만들지 않는다). 죄책감 유발 화법 금지 — 잊히는 것은 정상이다(Mind §1).

### B. 이해 (understanding)

**B1. "왜 저장했지" 패턴 찾기.** 흩어진 파편들 사이 반복되는 축을 대신 짚는다.
3주에 걸쳐 따로 던진 것들이 사실 한 덩어리라는 걸 유저 대신 본다.
- 클러스터 엔진(§6-2)의 출력. 근거 파편 ≥3개 인용 필수.
- 화법: "요즘 3주간 이 축이 반복돼" (관찰) — "너 이거 좋아하잖아" (규정) 금지.

**B2. 경향·관심 파악.** 유저의 최근 경향성/관심을 파악해서 찾아주거나 돕는다.
- **저장된 프로필이 아니라 실행 시점 쿼리다** (§2-1). 최근 90일 창, 감쇠 가중.
- **지속 관심사 vs 단발성** 구분은 계산으로 나온다:
  - 지속 = 클러스터의 증거가 3주 이상에 걸쳐 분포
  - 단발 = 7일 이내 버스트 후 소멸
- **진행중 프로젝트 파악**: Mind의 명시적 프로젝트 + 프로젝트로 안 묶였지만 뭉치는 암묵 클러스터
  ("진짜 프로젝트일 수도, 언젠가 해야지 리스트일 수도"). 암묵 클러스터는 추측 화법으로만 언급.
  유저가 "맞아 그거 하고 있어"라고 **직접 확인해주면** 그 진술은 증거로 저장 가능 —
  Rudy의 추론은 저장 금지, 유저의 자기 진술은 저장 가능. 이게 §2-1의 경계선이다.

**B3. 링크 소화.** 링크를 던지면 요약하고, 왜 저장했는지 의도를 추정하고, 다음 액션을 제시한다.
- 저장 시 비동기: 본문 수집(유튜브는 제목·설명·자막) → 3줄 요약 + 의도 추정 + 다음 액션 1개
- 의도 추정의 재료 = 같은 시기에 던진 파편들의 맥락 (혼자 있는 링크면 추정 생략 — §2-4)
- 표시 위치: 파편 상세. 푸시하지 않는다.
- 파생물(요약 등)은 캐시다 — 재생성 가능해야 하고, 원문과 구분되어 표시된다.

### C. 대화 (conversation)

**C1. 기억 기반 대화.** 파편을 다 훑지 않아도 물어보면 대답한다. 바로바로 알아듣고 대화한다.
- RAG: 질문 임베딩 → items 검색 → 근거 파편 인용과 함께 답변. **인용 없는 단정 금지.**
- 모르거나 확실하지 않으면 무조건 되물은 후 답변 (§2-4).
- 스트리밍 + 빠른 모델로 "바로바로" 체감 확보.
- **대화 중 자발적 연결 (킬러 무브).** 유저의 채팅 메시지마다 임베딩 → 코퍼스 충돌 검사 → 임계 통과 시
  "그거 3주 전에 던진 ○○랑 이어지는데?"를 답변에 엮는다. 미달이면 침묵(§2-8).
  묻지 않았는데 과거가 대화에 걸어 들어오는 순간 — 채팅이 검색창에서 "나를 아는 상대"로 바뀐다.
  GPT가 물리적으로 못 하는 것(코퍼스 부재). 이 연결도 원장에 기록(표면 간 중복 방지), touch 불변(§2-3).
- **원탭 진입.** 파편 상세에 칩("이거 관련 뭐 있었지" / "이거 다음 뭐 볼까") → 바로 채팅.
  파편 하나하나가 Rudy로 들어가는 문. 타이핑 마찰이 wow 사이의 평일 사용을 죽인다.
- (제안) 음성 입력 — "말을 알아듣고"를 문자 그대로 하려면 STT 붙이기. 선택 사항, §11에서 결정.

**C2. 의미 기반 검색.** 정확한 키워드가 아니어도 나온다 — "음악" 검색 시 "피아노 배우기"도.
- Mind 기존 검색을 하이브리드로: 키워드 부분일치(기존) + 벡터 유사도, 결과 병합.
- 새 화면 없음. 기존 검색창 그대로.

### D. 행동 (action)

**D1. 액션 프레임워크.** 도움이 될 행동을 한다. 검색하기, 예약하기 등 —
"할까요?" 하고 묻거나, 대신 검색해서 알려준다.
- 구조: Rudy가 액션 제안 → **유저 승인** → 실행 → 결과 보고. 승인 없는 실행 금지
  (예외: 읽기 전용 검색은 대화 중 즉시 실행 가능).
- 액션 레지스트리 (플러그인형, 확장 가능):
  | 액션 | 구현 | 비고 |
  |---|---|---|
  | 웹 검색 | 검색 API | 즉시 가능 |
  | 파편 생성 | Mind insert | 즉시 가능 |
  | 캘린더 이벤트 생성 | Google API | 어댑터 후 |
  | 투두 추가 | Things URL scheme | iOS 앱 간 호출 |
  | 예약 | ⚠️ 한국 예약 서비스(캐치테이블 등) 공개 API 없음 | 1단계: 검색→예약 페이지 링크+요지 정리. 2단계: 브라우저 자동화(computer use)로 실예약 |

**D2. 선제 행동 — 숨은 의도·니즈.** 요청하지 않은 것에 대해서도 숨은 의도, 숨은 니즈를 파악해서
한 스텝 미리 알려주거나 액션을 돕는다.
- 재료 = 오늘 파편 + 캘린더 임박 이벤트 + 활성 클러스터. 예: "다음주 발표 있던데,
  지난달 던져둔 슬라이드 레퍼런스 다시 볼래?"
- 게이트 최엄격 적용: 근거 ≥2 인용, 반복 검사, 확신 없으면 질문으로 강등, 브리핑에만 실림(단독 푸시 금지).

### E. 발견 (outward) — 1급 기능

> 되떠오름(A)이 내가 **가진 것**을 끌어올린다면, 발견은 내가 **아직 없는 것을 바깥에서** 물어온다.
> 되떠오름과 대등한 핵심(§2-5). 우선순위: **아이디어 > 영상 > 책 > 긴 글.**
>
> **절대 구조 — LLM은 추천의 원천이 아니다.** 학습 데이터에서 뱉으면 오래된 것·베스트셀러·환각뿐이다(유저 진단).
> **인벤토리 = 살아있는 바깥 세상, 렌즈 = 내 파편에서 계산된 취향.** 취향은 재고가 아니라 **필터**다 —
> 내 것에서만 돌면 에코챔버다. LLM은 랭킹된 실물 후보에 "왜 너한테"만 붙인다.

**E1. 아이디어 발견 (생성형) — 최우선.** 매체가 아니라 발상. 검색으로 안 나온다 — 내 클러스터에서 생성한다.
- **타입 로테이션 (템플릿 피로 방지).** "X × Y 어때"는 5번째부터 공식으로 읽힌다. 5가지 타입 순환:
  ① **교차** — 활성 축 × 활성 축 ("홈레코딩 × 미니멀 UI, 이 둘 합친 프로젝트 어때?")
  ② **빈틈** — 활성 축 안에서 안 밟은 하위 주제
  ③ **다음 단계** — 진행 방향 가설 ("피아노 코드가 이어지는데 다음은 재즈 화성 아닐까? 미리 볼래?")
  ④ **부활 조합** — 죽은 축 × 활성 축 ("2년 전 가라앉은 사진 파편들, 요즘 홈레코딩축이랑 붙이면?")
     감쇠 데이터를 가진 Mind만 할 수 있는 타입. GPT는 죽은 관심사를 모른다. 이 시스템의 유일무이한 무기.
  ⑤ **반론** — 축이 전제하는 것의 반대편
- 빈도: **주 1개 수준(§2-8).** 아이디어는 희소해야 아이디어다.
- 생성이지만 **내 파편에 근거**하고 **추측 화법 강제**("이건 내 추측인데"). §2-1이 가장 세게 시험받는 기능 —
  화법 게이트 위반 시 발화 폐기. 틀리면(무시·거절) 그 축 백오프(§6-5).

**E2. 미디어 발견 (검색형) — 영상·책·긴 글.**
- **소스를 고정하지 않는다. 실행 시점에 찾는다.** 활성 축의 구체 명사 → "이 주제는 지금 어디서
  신선하게 도나"를 그때그때 판단해 검색·수집한다. 소스 화이트리스트조차 저장하지 않는다(§2-1 정신).
- 스코어 = sim(축) × 소스품질 × 신선도 − 이미저장페널티 × 매체다양성. novelty는 바깥 것이라 기본 확보.
- **와일드카드 1개는 축을 끈다** — 순수 세렌디피티(F2의 발견 버전). 렌즈만 걸면 취향도 좁아진다.
  이게 "재미"의 안전장치. ⚠️ 와일드카드는 **근거 게이트(§6-4 ①) 면제** — 인용할 파편이 없는 게
  정의 그 자체다. 대신 와일드카드임을 표시하고 나간다 ("이건 네 축이랑 무관해. 그냥.").
- **한·영 이중 탐색.** 파편은 한국어가 대부분이지만 축의 최전선 콘텐츠는 영어권에 있는 경우가 많다.
  장소 탐색(①)은 한국어·영어 양쪽을 본다.
- **에이전틱 루프 — 원샷 검색 금지.** 검색 API 한 방 = SEO 파밍 결과(유저가 겪은 쓰레기의 원인).
  ① "이 축은 지금 어디서 살아있나" 장소 탐색 → ② 그 안에서 검색 → ③ 수집·평가·필터.
  발견 품질의 실제 승부처. 품질이 나쁘면 구조를 갈지 말고 ①만 튜닝한다 (실패의 격리).
- **리듬: 주 2~3회 배치 생성, 3~5개, 빈 피드 허용(§2-8).** 매일 채우지 않는다.
- **렌즈 환류.** 피드에서 열었거나 던진 것의 스타일·매체·출처 경향을 다음 랭킹에 가중 —
  원장에서 실행 시점 계산 (§2-1 무저장 정합).

**E3. 큐레이션 품질 (유저 진단 정면 대응).**
1. *쿼리가 리스트성 글을 부른다* → 축의 구체 명사로 쿼리, `best/top/추천/N가지` 금지, 리스티클 제목 필터.
2. *어디서 가져오는지가 근본* → 인벤토리는 열린 바깥, 렌즈는 내 취향. 1차 소스 > 애그리게이터.
   내 저장 소스는 아이템 원천이 **아니라** "내가 어떤 스타일을 신뢰하나"의 약한 가중 신호로만.
- 원장(§5)으로 중복 추천·같은 소스 과다 노출 원천 차단.

**E4. 던지기 플라이휠.** 발견 아이템에 **"던지기" 버튼 → 그대로 Mind 파편이 된다** (공유시트와 같은 경로).
발견이 소비로 끝나는 막다른 골목(모든 추천 앱이 죽는 지점)이 아니라 **기억의 재료로 순환**한다:
던진 발견 = 최강의 acted 신호 + 코퍼스 풍부화 + 다음 충돌·클러스터에 참여.
되떠오름과 발견은 별개 기능이 아니라 하나의 루프다.
- 피드 응답: 던지기(acted) / 넘기기(dismissed) / 무시(ignored, 무벌점 §6-5).
- 던져진 파편의 metadata에 출처(discovery)만 표시 — 사실 기록이지 해석 아님. 성적표(§6-6) 집계용.

### F. Rudy의 제안 기능 (유저 리스트에 없던 것, 이 문서에서 제안)

**F1. 늦은 의도 붙이기.** 저장 시점엔 아무것도 묻지 않지만(§2-7), 나중에 대화 중 자연스럽게
"어제 던진 그 링크, 뭐 하려던 거야?"라고 가볍게 묻는다. 유저의 답 = 최고 품질 증거(자기 진술).
답 안 해도 아무 일 없음.

**F2. 랜덤 주입 (반-룹 장치).** Rudy가 모든 되살리기를 장악하면 활성 클러스터만 강화되는
피드백 룹이 생긴다. 떠오른 것 하루 2개 중 1개는 **영원히 순수 랜덤**으로 남긴다.
Rudy가 절대 건드리지 못하는 우연의 자리. 헤매기도 그대로 랜덤 유지.

**F3. Rudy 출력물의 감쇠.** Rudy의 통찰·제안도 파편과 같은 법을 따른다 — 유저가 무시하면
흐려지고 반복되지 않는다. 이 시스템의 모든 것은 잠정적이며, 유저의 판단으로만 영속성을 얻는다.

**F4. 아침 브리핑 (전달 모델).** Rudy가 먼저 거는 말의 묶음. 오늘의 캘린더·투두 맥락 +
맥락 리서페이스(A2) + 넛지(A3) + 선제 관찰(D2) 중 게이트를 통과한 항목. 볼 것 없으면 안 보낸다
(빈 브리핑 금지). 빈도·전달 채널은 유저가 조정(§7-3).
푸시는 결정됨(§11-3) — 하루 1회 브리핑만 허용(§7-3). 주 1회는 브리핑 대신 주간 편지(F5)가 실린다.

**F5. 주간 편지 (리추얼).** 주 1회, Rudy가 짧은 **편지**를 쓴다 — "이번 주는 ○○의 주였네.
화요일에 던진 A가 금요일의 B랑 이어지더라. 두 달 전 가라앉은 C가 이번 주 흐름과 닿아 있어."
- **대시보드가 아니라 서사.** 숫자·통계·달성률 금지 (Mind 생산성 문법 배제 상속). 관찰만.
- "이번 주의 너"는 규정이 아니라 스냅샷 — 시간 한정 화법 그 자체 (§2-1 정합).
- 인간은 자기 일주일의 모양을 스스로 못 본다. Rudy는 본다 (Wrapped가 사랑받는 이유 = 자기 자신을 보여줘서).
- 브리핑(정보)과 역할이 다르다 — 이건 **거울**이다. 주 1회 희소성이 "기다림"을 만든다 — 지속 사용의 리추얼 축.
- 재료가 빈약한 주엔 안 보낸다(§2-8). 전달은 새 푸시 채널이 아니라 **그날의 브리핑 슬롯**에 실린다(§7-3 정합).

## 5. 메모리 설계

**저장하는 것 (전부 사실 또는 기록):**

```
rudy_utterances (원장 — §2-2의 물리적 실체)
  id, created_at, surface (briefing|chat|recall_feed),
  kind (resurface|nudge|pattern|discovery|prediction|question|action_proposal),
  medium (idea|video|book|article|null),  -- 발견류 백오프의 축×매체 단위(§6-5)용
  item_ids[]          -- 근거로 인용한 아이템들
  text, embedding,    -- 발화 자체를 임베딩 (반복 검사용)
  user_response (acted|dismissed|ignored|null), responded_at

rudy_evidence (유저의 자기 진술만)
  id, created_at, stated_text, related_item_ids[], embedding
  -- 예: "맞아, 홈레코딩 준비하고 있어" — 유저가 직접 말한 것만 들어온다

rudy_actions
  id, utterance_id, action_type, status (proposed|approved|executed|declined),
  payload jsonb, result jsonb, executed_at

conversations / messages   -- 채팅 이력 (대화 맥락용)

link_digests               -- B3 캐시 (재생성 가능한 파생물)
  fragment_id, summary, guessed_intent, suggested_action, created_at
```

**저장하지 않는 것 (의도적 부재 — 이것이 §2-1의 물리적 실체):**
- ❌ `user_profile` / `user_interests` / `user_traits` — **이 테이블들이 없는 것 자체가 설계다.**
- 관심사·경향·암묵 프로젝트는 전부 실행 시점 계산. 성능용 캐시는 TTL 24h까지 허용하되
  캐시는 언제든 버리고 재계산 가능해야 한다 (캐시 ≠ 결론).

**하나의 대칭:** 임베딩 하나가 시스템 전체를 관통한다 —
파편 충돌(A1), 의미 검색(C2), 클러스터(B1·B2), 반복 억제(§2-2), 큐레이션 중복 차단(E1)
전부 같은 벡터 연산이다.

## 6. 엔진

**6-1. 충돌 엔진** — A1 상세는 §4-A1. 새 파편 임베딩 완료 시 트리거.

**6-2. 클러스터 엔진** — B1·B2의 심장.
- 입력: 최근 90일 items, 선명도 가중 (흐려진 증거는 약하게 반영 — 감쇠 상속)
- 클러스터링은 실행 시점, 결과는 휘발성(TTL 캐시만). 라벨은 LLM이 그때그때 붙인다.
- 지속성 척도 = 증거 타임스탬프 분포 (§4-B2).

**6-3. 컨텍스트 트리거** — A2·F4. pg_cron 스케줄(아침 브리핑) + 캘린더 폴링 + 지오펜스 이벤트.

**6-3b. 발견 엔진** — §4-E의 심장. 실행 시점에:
(a) 활성 축의 구체 명사 추출 → (b) **소스를 그때 찾아** 검색·수집(고정 리스트 없음) →
(c) 취향 렌즈로 랭킹 + 이미저장 페널티 → (d) 와일드카드 1개 주입 → 발화 파이프라인.
소스도 결과도 저장 안 함(TTL 캐시만). 아이디어(E1)는 클러스터 교차를 LLM이 생성하는 별도 경로.

**6-4. 발화 파이프라인 — 모든 발화형 출력의 관문.**
```
후보 생성 (각 엔진)
 → ① 근거 게이트: 인용 item ≥1 (패턴·예측류는 ≥2~3). 인용 못 하면 폐기
 → ② 반복 게이트: 원장 90일 내 유사 발화(cos > 0.85) 존재 → 폐기 또는 새 각도로 재작성.
    cos는 근사 중복만 잡는다 — "다른 말로 같은 소리"는 같은 축 과거 발화를 뽑아 LLM 판정(Haiku)으로 보강
 → ③ 화법 게이트: 시간 한정 필수, 정체성 단정 패턴 검출 시 폐기, 관찰형/질문형 강제.
    한국어 단정 화법("너 원래 ~잖아")은 정규식으로 못 잡는다 — 정규식 + LLM 판정 겸용
 → ④ 확신 게이트: 불확실 → 단정문을 질문문으로 변환 (§2-4)
 → ⑤ 예산: 브리핑 1회/일·최대 3항목, 떠오른 것 공급 1개/일, 채팅 응답은 무제한
 → 전달
```
(수치는 전부 초기값 — 실사용으로 튜닝한다. **모든 게이트 판정은 사유와 함께 로그** —
임계 튜닝은 감이 아니라 이 로그로 한다.)

**6-5. 반응 학습 (백오프).** 같은 축의 발화가 2회 연속 dismissed → 그 축 14일 침묵.
단위는 **축×매체** — 홈레코딩 영상 2개 넘겼다고 홈레코딩 글까지 죽이면 과잉이다.
축은 저장되지 않으므로(휘발성 클러스터) "같은 축" 판정은 **과거 발화 임베딩과의 근접(cos)**으로 한다.
ignored는 벌점 없음 (Mind §5-1 "참여 안 해도 벌이 없다" 상속). acted → 해당 축 신뢰 유지.
이것은 유저 모델이 아니라 **Rudy 자신의 행동 조절**이다 — 원장만으로 계산된다.

**6-6. 월말 성적표.** 원장에서 실행 시점 집계한 기능별 acted/dismissed/ignored 월간 리포트.
"내가 가치를 느끼나"를 감이 아니라 데이터로 — 어떤 기능을 죽이고 어디를 튜닝할지 유저가 판단할 근거.
- **응답 캡처 정의 (성적표·백오프의 전제):** acted = 던지기·링크 열기·기억하기·액션 승인 /
  dismissed = 명시적 넘기기·흘려보내기 / ignored = 노출 후 무반응. 노출 자체는 어떤 신호도 아니다(§2-3).

## 7. 표면 (유저가 Rudy를 만나는 곳)

**7-1. Mind 떠오른 것 공급 (조용한 표면).** 데일리 뷰 하루 2개 중 1개를 충돌 엔진이 공급 —
단 **임계 통과 시만**, 못 넘는 날은 랜덤 2개(§4-A1). Mind의 규칙 그대로: 큐·개수·진행률 없음,
노출≠touch, 기억하기/흘려보내기/무시. 기본은 조용하다 — 단 **탭/롱프레스하면 "왜 지금"이 읽힌다**
(§4-A1 요청 시 가시성). 조용한 개선 + 요구하면 보이는 마법.

**7-2. 채팅 (당기는 표면).** C1·C2·D1·F1·B1~E2 요청형 전부. Mind 앱 안 탭으로 시작
(권장 — 이유: 유일한 유저가 이미 매일 여는 앱, auth·데이터 접근 재사용).
단 **Rudy 백엔드는 처음부터 분리 설계**(§8) — 나중에 별도 앱·데스크탑·음성으로 표면을
옮겨도 코드가 사는 구조.

**7-3. 미는 표면 (브리핑·편지).** F4·F5. **푸시 정책은 열어둔다 (결정 2026-07-19) — 유저가 실사용 후 조정.**
하루 1회로 고정하지 않는다. 다만 스팸이 되면 죽는다는 것만 기억하고(§2-8 침묵 기본값은 여전히 유효 —
말할 가치 없으면 안 보냄), 빈도·채널은 유저가 쓰면서 튜닝한다.

**7-4. 발견 피드 (당기는 표면).** §4-E 전부의 집. 내가 열고 싶을 때 여는 **개인 매거진** — 그레이징(훑어보기)용.
아이디어·영상·책·긴 글 + 와일드카드가 매체 다양성으로 섞여 뜬다. 되떠오름 슬롯(7-1)과 분리 —
그건 조용한 회상, 이건 능동적으로 노는 곳("재미"). 스팸 방지: 당기는 표면 + 브리핑 1개, 산발 푸시 금지.
아이템마다 **던지기 / 넘기기**(§4-E4 플라이휠). 주 2~3회 배치 생성, 빈 피드 허용(§4-E2).
가끔 **보이는 거절** 한 줄(§2-b) — 필터가 일하는 게 보이면 통과한 것의 가치가 올라간다.

## 8. 아키텍처

```
[Mind 앱 (Expo)] ─── 채팅 탭 · 떠오른 것 · 브리핑 카드 · 파편 상세(링크 다이제스트)
       │
[Supabase (Mind와 같은 프로젝트, `rudy` 스키마 분리)]
   pgvector          — item_embeddings, 발화 임베딩
   pg_cron           — 브리핑 스케줄, 캘린더 폴링, 임베딩 백필
   Edge Functions    — embed(트리거), chat(스트리밍), briefing, collide, digest, act
       │
[외부] OpenAI API (임베딩 + 대화) · 검색 API · Google Calendar
[맥]   Things 브리지 스크립트 (launchd 주기 실행 → Supabase로 push)
```

- **Mind SPEC §3 "서버 코드 직접 작성 없음"과의 관계:** Mind 앱은 그대로 무서버.
  Rudy는 층 3이므로 Edge Functions를 가진다 — 단 별도 서버 운영은 없다 (전부 Supabase 안).
- **모델:** 임베딩 = OpenAI `text-embedding-3-large` (3072차원, 확정 2026-07-19).
  대화 = **OpenAI Chat Completions** (확정 2026-07-19 — 임베딩과 한 공급자로 통일).
  모델 ID는 코드가 아니라 Edge Function env `OPENAI_CHAT_MODEL`에 있다(기본 `gpt-4o`) —
  지연·품질 보고 바꿀 때 배포 없이 값만 갈아끼운다. 유저 1명 기준 월 비용 수 달러 수준.
- **키 관리:** API 키는 Edge Function 환경변수. 앱에 키 내장 금지.

## 10. 개발 순서 (순수 의존성 순 — 전부 목표, 미루는 것 없음)

```
0. `rudy` 스키마 + pgvector + 임베딩 파이프라인 + 전체 백필   → 모든 것의 토대.
   시작 전 **임베딩 모델 2종을 실제 내 파편 검색으로 비교 후 선택** (한국어 품질이 시스템 전체의 바닥이다)
1. 링크 다이제스트 + 링크 파편 재임베딩 (B3, §3-2)             → 임베딩 인프라의 일부. 없으면 링크 파편 전원 유령
2. 의미 검색 (C2)                                            → 품질 리트머스
3. 충돌 엔진 → 떠오른 것 공급 + 요청 시 가시성 (A1, 7-1)       → 임계 게이트 포함, 억지 충돌 금지
4. 원장 + 발화 파이프라인 + 게이트 판정 로그 (§5, §6-4)        → 이후 모든 발화형 기능의 관문
5. 채팅 표면 + RAG + 자발적 연결 + 원탭 진입 + 즉시 검색 (C1, 7-2, D1 일부)  → 원장 필요
6. 클러스터 엔진 → 패턴·경향 (B1, B2) + 늦은 의도 (F1)         → 채팅 표면 필요
7. 발견 엔진: 아이디어 로테이션·에이전틱 미디어 검색·플라이휠 + 발견 피드 (§4-E, 7-4)  → 클러스터+원장
8. 브리핑 + 넛지 + 주간 편지 + 월말 성적표 (F4, A3, F5, §6-6)  → 파이프라인 + 클러스터
9. 어댑터: 구글 캘린더 → Things 맥 브리지 (§3-1)               → items 정규화 확장
10. 컨텍스트 트리거 (A2: 시간·계획·장소)                        → 어댑터 필요
11. 액션 확장: 캘린더 생성·투두 추가·예약 1단계 (D1)            → 어댑터 + 승인 UI
12. 선제 행동 (D2)                                            → 전체 기계가 검증된 위에서만 안전
```

각 단계 완성 = 실사용 가능한 조각. 12까지가 이 문서의 전체 범위다.
