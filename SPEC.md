# SPEC.md — 프로젝트 기획서 (개발 착수 전 1회 정독)

## 1. 이 앱이 무엇인가

북마크앱도, 투두앱도, 생산성 앱도 아니다.
**사용자(단 한 명)의 기억과 뇌를 흉내내는 아이디어 파편 저장소**다.

사용자의 행동 패턴: 아이디어 파편이 수시로 떨어진다. 놓치기 싫어서 던져두지만
다시 잘 보지 않는다. 평생 정리를 하지 않는 사람이며, 정리를 요구하는 앱은 반드시 죽는다.
문제는 저장도 검색도 아니다 — **전부 남기니까 전부 안 보이는 것.**
파편이 같은 무게로 평평하게 쌓여 벽이 되는 것이 유일한 병목이다.

해결 원리 — 기억처럼 작동시킨다:

> **파편은 던진 순간 선명하고, 시간이 지나면 흐려지고, 건드리면 다시 선명해진다.**

잊히는 것은 정상이고 건강한 것이다. 안 보는 행위 자체가 판정이다.
사용자가 아무것도 정리하지 않아도 시스템은 완전히 작동해야 한다.

## 2. 절대 원칙 (구현 결정과 충돌 시 이쪽이 이긴다)

1. **입력 마찰 0** — 저장 시점에 어떤 결정도 요구하지 않는다.
   타입 선택·프로젝트 선택·태그 전부 금지. 자동이거나, 나중이거나.
2. **생산성 문법 배제** — 완료 체크, 달성률, 스트릭, 마감일, 알림 없음.
   태스크 도구가 아니라 자기 관찰 도구다.
3. **폴더/태그 없음** — 시간축과 선명도가 구조를 대신한다.
4. **프로젝트는 폴더가 아니라 필터(렌즈)다** — 같은 파편이 전체 타임라인에도,
   프로젝트 뷰에도 보인다.
5. **사용자는 한 명** — 멀티유저 고려, 온보딩, 일반화 금지. 과잉 추상화 금지.
   엔터프라이즈 패턴(DI 컨테이너, 과도한 모듈 분리) 도입 전에 "1인용 앱에 필요한가"를 물을 것.

## 3. 스택

- **iOS 앱**: Expo (React Native + TypeScript)
- **서버**: Supabase (Postgres + Auth + Storage + REST) — 서버 코드 직접 작성 없음, 무료 티어
- **공유 저장 (expo-share-intent)**: 필수. 이 앱의 심장. 사파리/유튜브/사진 어디서든 2탭 저장
  (공유 → Mind 선택 → 앱 열리며 자동 저장). 이게 안 되면 앱 전체가 무의미하므로 최우선 구현 대상

~~**웹 입력 페이지**~~ — **폐기 (2026-07-14).** 회사 데스크탑에서 던지기용으로 계획했으나,
브라우저 탭을 열어 붙여넣는 마찰이 남는다. 데스크탑 입구가 필요해지면 웹 페이지가 아니라
**크롬 익스텐션이나 맥 공유 시트** 쪽으로 간다 — 그쪽이 "입력 마찰 0"에 실제로 부합한다.

## 4. 데이터 모델

**Fragment**
| 필드 | 설명 |
|---|---|
| id, created_at | 기본 |
| content | 원문 |
| type | text / link / image / quote — **자동 판별** |
| link_title, link_thumbnail_url | link일 때 비동기 수집 |
| image_path | image일 때 (Supabase Storage) |
| last_touched_at | 선명도 계산의 입력값 |
| tier | normal / important / pinned. **pinned = 즐겨찾기** (별도 개념을 만들지 않는다) |
| touch_count | **회상에서 구해낸 횟수 = 자라나는 중요도** (2026-07-14 추가) |
| let_go_at | **회상에서 흘려보낸 시각.** 보여준 것만으론 기록되지 않는다 (2026-07-14 추가) |
| project_id | nullable. null = inbox. 사용자가 나중에 붙임 |
| archived | 바닥 여부 |

타입 자동 판별: URL 패턴 → link, 이미지 파일 → image, 인용부호/출처 형태 → quote, 나머지 → text.

**선명도는 저장하지 않는다.** last_touched_at + tier로 뷰 레이어에서 계산.
데이터는 사실만, 해석은 화면에서.

**Project** — 파편과 다른 종류의 아이템. 타임라인에 쌓이지 않고 폴더처럼 옆에 선다.
| 필드 | 설명 |
|---|---|
| id, name | 기본 |
| status | before / active / paused / done |
| started_at | 시작일 (프로젝트의 유일한 시간축) |
| description | 이게 뭔지에 대한 자유 서술 |

파편↔프로젝트는 **다대다** — 한 파편을 여러 프로젝트에 태그처럼 붙일 수 있다 (2026-07-12 개정).

여기까지다. **태스크·마감일·진척률·완료 체크는 여전히 이 앱의 적이다.**
프로젝트는 "지금 내 삶에서 뭐가 살아있나"를 보는 지도일 뿐, 관리 도구가 아니다.

## 5. 선명도 규칙 (확정값)

| tier | 감쇠 시작 | 바닥(투명도 15%) 도달 |
|---|---|---|
| normal | 1일 | 7일 |
| important | 7일 | 21일 |
| pinned | 감쇠 없음 | — |

**2026-07-22 개정 — 이전 값은 `3/14`·`14/45`·바닥 25%였다.** 실사용에서 "애초에 흐려지질
않는다"가 나왔고 실측이 그걸 확인했다: 파편 127개 중 **바닥에 닿은 게 0개**, 제일 오래된
것도 0.76이라 100%와 눈으로 구분이 안 됐다. 더 중요한 건 **회상 후보(선명도 ≤ 0.7)가
0개**였다는 것 — §5-1이 통째로 굶고 있었다. 새 값에서 후보는 69개가 된다.

- **여는 것만으로는 last_touched_at이 갱신되지 않는다.** 내용·덧붙임을 실제로 고칠 때, 또는
  **프로젝트를 지정/해제할 때** 갱신된다 — 프로젝트에 다시 붙이는 건 "지금 이걸 다시 붙잡았다"는
  판단이다(2026-07-22 유저 확정). **tier 변경만은 touch가 아니다** — tier 자체가 감쇠 속도를
  바꾸므로 touch까지 하면 같은 신호가 중복된다.
- **흐려지는 걸 멈추는 명시 행동:** 내용 수정, 프로젝트 지정/해제, 회상의 `기억하기`,
  파편 상세의 `기억하기` 버튼(흐려진 것에만 뜬다). 전부 100% 복귀 + touch_count +1.
  단순히 열어보는 것으로는 절대 살아나지 않는다.
- 바닥 도달 후에도 리스트에 남는다(15% 고정). 삭제는 수동만.
  단 상세 화면은 감쇠를 적용하지 않고, `wander`는 읽을 수 있게 0.55로 클램프한다 —
  **흐림은 피드의 표현이지 접근 차단이 아니다.**
- 감쇠 곡선은 선형으로 시작
- **두 번 이상 구해낸(touch_count ≥ 2) normal은 important 곡선을 탄다.**
  중요도를 손으로 정할 필요가 없다 — 자라난다. 수동 tier는 그대로 우선한다.

## 5-1. 회상 (2026-07-14 추가 — 이것이 없으면 §5는 소멸일 뿐이다)

**진단:** 선명도는 last_touched_at으로 계산되는데 타임라인은 created_at 순이다.
사용자가 옛 파편을 열지 않으면 둘은 영영 같고, 그러면 선명도 = 최신성 = 스크롤 위치 →
**fade가 아무 정보도 담지 않는다.** 그런데 §1은 사용자를 "던져두고 다시 안 보는 사람"으로 정의한다.
감쇠는 만들고 회상을 안 만들면, 그건 기억이 아니라 그냥 잊힘이다.

**절대 규칙: 그냥 봤다고 선명해지면 안 된다.**
시스템이 들이민 노출은 아무 신호가 아니다. 회상 화면에서는 **노출이 아니라 판단이 touch**다.

**떠오른 것** (데일리 뷰, 오늘을 볼 때만, 하루 2개) — **판단하는 자리.**
- **기억하기** → 선명도 100% 복귀 + touch_count += 1
- **흘려보내기** → 파편엔 아무 일도 없다. 계속 흐려진다. 60일간 다시 안 뜰 뿐.
  **삭제도 아카이브도 아니다** — 그러면 §7이 금지한 솎아내기 노동이 된다.
- **무시(안 누르고 나감)** → 아무 흔적도 안 남는다. 나중에 또 뜰 수 있다.
  참여하지 않아도 벌이 없어야 §1의 "안 보는 행위 자체가 판정이다"가 산다.

**후보 선정** — 잊히기 직전(선명도 0.7 이하, 2026-07-19 완화 — 0.5는 normal tier 기준
10일+ 미접촉을 요구해 실사용 초기 후보가 비었다)인 것만. 이미 또렷한 걸 또 보여주는 건 낭비다.
회상의 가치는 *잊고 있었는데 다시 만나는 것*에 있다.
그 안에서 tier·touch_count가 높을수록 뽑힐 확률이 높다 → **중요한 게 잊히고 잡스러운 게 남는 문제**가 풀린다.
**큐·개수·진행률 표시 금지.** 넣는 순간 생산성 앱이 되고 죽는다.

**헤매기** (사이드바, 무한 랜덤) — **판단하지 않는 자리.**
- **기억하기/흘려보내기 버튼이 없다.** 무한히 뽑을 수 있는 곳에 판단을 붙이면
  보관함 전체를 솎아내는 노동이 된다 — §7이 금지한 정리 스와이프다.
- 지나가면 아무 일도 없다. 탭해서 열면 스스로 찾아간 것이므로 그때 touch된다.
- 나란히 흘러가는 파편들이 곧 병치다. **연결은 저장하지 않는다** (§7 참조).

## 6. 화면 (MVP)

0. **데일리 뷰 (기본, 2026-07-12 추가)** — 상단 주간 캘린더 스트립(날짜 밑에 파편 밀도 점,
   점의 투명도 = 선명도 → 캘린더가 곧 기억의 지도), 아래에 그 하루의 파편이 불릿 한 줄씩.
   던지기는 보고 있는 날짜와 무관하게 언제나 오늘로. 상세는 PLAN.md §6.1

1. **타임라인 피드** — 카드 단위, 최신 위, 날짜 구분선.
   선명도를 카드 opacity로 시각화 — 스크롤 내려 과거로 갈수록 연해지는 지층감이 이 앱의 정체성.
   상단에 필터 칩 (All + 진행중 프로젝트만). Inbox·무덤·전체 프로젝트 목록(상태별)은
   좌측 사이드바(엣지 스와이프로 열고 닫음)에서 접근. 파편은 탭→상세, 우→좌 스와이프→수정/삭제
2. **어젠다 뷰** — 날짜가 섹션 헤더(큰 숫자+요일), 아래 파편이 한 줄 행으로 밀도 높게.
   훑기용. opacity 감쇠 동일 적용. 피드와 상단 토글로 전환
3. **빠른 입력** — 붙여넣기/타이핑 → 타입 자동 인식 배지 → 던지기.
   프로젝트 선택은 접힌 옵션(기본 Inbox)
4. **파편 상세** — 원문 전체. tier 3단 토글, 프로젝트 붙이기, 삭제. 여는 것만으로는 touch 안 됨 —
   내용·덧붙임·tier·프로젝트를 실제로 바꿀 때만 touch(2026-07-19 개정, §5 참조)
5. **공유 저장** — 공유 시트에서 Mind 선택 → 앱 열리며 자동 저장 + 토스트. 프로젝트 선택 넣지 말 것
6. **떠오른 것** — 데일리 뷰 안, 오늘 파편들 아래. **별도 화면으로 만들지 않는다** —
   가야 하는 곳이 되는 순간 안 가게 된다. 이미 가는 곳에 놓는다. 상세는 §5-1
7. **헤매기** — 무한 랜덤. 판단 버튼 없음. 상세는 §5-1
8. **사이드바 렌즈** — All · Inbox · **고정**(= tier pinned) · 프로젝트 / 헤매기 · 무덤

세부 스타일은 design.md를 따른다.

## 7. 만들지 않는 것 (지금 언급도 구현도 하지 말 것)

메이슨리 뷰, **정리 스와이프(솎아내기)**, **파편 간 연결을 저장하는 것**(브레인맵),
macOS 네이티브 앱, 위젯

**경계선 — 오늘 논의로 겨우 얻은 선이므로 지킬 것 (2026-07-14):**
- **병치는 하되 연결은 저장하지 않는다.** 무관한 파편 둘을 나란히 놓는 것은 좋다(§5-1 헤매기).
  링크를 *저장*하는 순간 그것을 관리하는 노동이 생기고, 정리를 요구하는 앱은 반드시 죽는다(§1).
- **회상의 판단은 솎아내기가 아니다.** 하루 2개, 큐 없음, 안 해도 아무 일 없음 —
  이 셋 중 하나라도 깨지면 그때부터 정리 스와이프다.
- **헤매기에 판단 버튼을 추가하자는 제안은 이 조항으로 거절한다.**

**해제된 항목:**
- 검색 — v2 → MVP 편입 (2026-07-12). 원문·링크 제목 부분일치, 무덤 포함
- 날짜 점프 — 구현함 (2026-07-14). 피드/어젠다 스티키 날짜 pill → 월 캘린더 → 해당 섹션으로
- AI 레이어 — 회상이 실사용으로 검증된 뒤 착수 (§5-1이 없으면 AI는 "안 보는 더미에 요약 하나"일 뿐이다)
- 라이트 모드 — 하기로 함 (2026-07-14). 우선순위는 뒤

⚠️ **방향 경고:** 지금까지 만든 것 대부분이 *저장*과 *탐색*이다(피드·어젠다·데일리·검색·캘린더·프로젝트).
SPEC이 말하는 본질은 탐색을 늘리는 게 아니라 **떠오르게 하는 것**이다.
새 기능을 제안할 때 **"이건 탐색을 늘리나, 회상을 늘리나"** 를 먼저 물을 것.
